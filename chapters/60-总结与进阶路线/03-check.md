# Chapter 60 总结与进阶路线 - 自测与验收

## Q1 概念: 写简历项目描述的"原子单位"是什么? 反面教材?

**原子单位**: 一条亮点 = **背景 + 改动 + 数字收益 + 关键文件**

```
✓ 好: 详情接口 P99 从 612ms 优化到 3ms (200x), 通过 Caffeine + Redis 双层
       缓存 + N+1 修复 + 加索引, DB 流量下降 92%。
       (Path: service/PostService.java#getDetail)

✗ 差: 使用 Redis 缓存提升了性能。
✗ 差: 熟练掌握 Spring Boot, 精通 MySQL。
✗ 差: 负责后端开发, 完成了多个功能。
```

**反面教材的 4 个特征**:
1. 没有数字: "提升了性能" — 提升多少? P99 从多少到多少?
2. 没有上下文: 为什么要做? 不做有什么问题?
3. 没有 trade-off: 改动有没有副作用? (如双缓存增加内存)
4. 没有代码定位: 面试官想看源码时找不到

## Q2 概念: 项目从 GitHub 到面试官 30 秒看完, 你的 README 必须有哪些?

**面试官 30 秒扫 README 决定深入与否**, 必须有:

| 块 | 内容 | 时间 |
|---|---|---|
| 一句话 | "做什么 + 怎么部署" (15 秒看完) | 0-15s |
| 技术栈表格 | 一眼看到用了什么 | 15-20s |
| 核心能力 | 3-5 个 bullet, 体现复杂度 | 20-25s |
| 快速开始 | `docker compose up -d` 一行 | 25-30s |

**不要写**:
- 长段落 (没人读)
- "本项目使用了 SpringBoot 框架开发..." (废话)
- 截图占 80% (没干货)
- 没架构图也没思路

**示范 (一句话)**:

> "Spring Boot 3 + Java 21 的博客后端, 涵盖认证 / 缓存 / 异步 / 文件 / 监控 / 部署
>  全链路, 一键 `docker compose up -d` 1 分钟起。 QPS 峰值 9200, P99 < 5ms。"

## Q3 实操题: 写 3 条简历亮点 (你自己的项目, 不准抄)

**模板**:

```
### 标题: 数字 + 关键技术

- 背景: 问题是什么 (1-2 句)
- 改动: 用了什么技术 / 思路 (1-2 句)
- 收益: 改前 vs 改后, 带数字
- 关键文件: src/main/.../XxxService.java
```

**写完 self-check**:
- [ ] 标题里有具体数字吗? (P99 / QPS / 容量 / 时间)
- [ ] "我做了什么" 而不是 "项目用了什么"?
- [ ] 一个面试官能问 5 个深入问题吗? (说明有深度)
- [ ] 是不是堆术语没逻辑? (用 → 收益 → 取舍)
- [ ] 关键文件路径可定位代码?

**例子 (反面 → 正面)**:

```
反面: "使用 Redis 实现了缓存, 提升了系统性能。"

正面: "热点文章详情接口 (10w QPS) 通过 Caffeine L1 (1us 访问) +
      Redis L2 (1ms) 双层缓存, 防穿透 (cache empty marker) +
      防击穿 (Redisson 分布式锁) + 防雪崩 (TTL ±10% 抖动),
      命中率 99.2%, DB 流量下降 92%, P99 从 80ms 降至 4ms。
      源码: cache/L2CacheManager.java"
```

## Q4 实操题: 模拟答辩 — 面试官 90 秒讲项目, 怎么组织?

**STAR + 数字 + 取舍** 公式:

```
[S 15s] 背景:
    "我做了一个完整的博客 API, 用 Spring Boot 3 + Java 21,
    目的是验证自己从 0 到 1 交付生产级后端的能力。"

[T 10s] 任务:
    "目标是一键启动 + P99 < 50ms + 测试覆盖率 60% + 简历可写。"

[A 50s] 行动 (挑 3 个最亮的):
    "1. 性能上, 详情接口我做了 N+1 + 双层缓存 + 索引优化,
        P99 从 612ms 降到 3ms;
     2. 测试上, 单元 + Testcontainers 集成, 覆盖率 65%,
        CI 强制阈值;
     3. 部署上, 多阶段 Dockerfile + GitHub Actions CI +
        Trivy 扫描 + 滚动部署脚本, 部署时间从 20 分钟到 90 秒。"

[R 15s] 结果 + 取舍:
    "QPS 峰值 9200, 镜像 250MB, 60 章笔记 + 5 道系统设计稿都开源。
    项目代码在 GitHub: <url>。
    主要 trade-off 是双层缓存增加 200MB 内存, 一致性靠 TTL + 失效保证,
    对博客这种读多写少 + 容忍秒级延迟的场景是合适的。"
```

**讲完面试官常问**:

1. 详情接口怎么优化的具体讲? (→ 准备 Caffeine 配置 / Redis Lua / 索引 EXPLAIN)
2. 怎么测? (→ 拿出 PostServiceTest 给他看)
3. 上过生产吗? (→ 没有, 但 docker-compose 跑过完整压测, 数据来自 wrk)
4. 哪里你觉得做得不够好? (→ 准备 2-3 个真实短板, 不要装完美)

**最后一问最关键**: "你觉得哪里做得不够?"

> 错答: "都挺好的"
> 对答: "我没有真实分布式经验, Spring Cloud / Seata 只是 demo 过; 系统设计
>         5 道题练过但临场可能慌; 大数据相关 (Kafka Streams) 完全没碰过。
>         这是我下半年要补的方向。"

## Q5 综合: 学完 60 章, 接下来怎么进阶?

### 3 个月路线 (深化已学)

| 方向 | 任务 | 验收 |
|---|---|---|
| 分布式 | Spring Cloud Gateway + Nacos + OpenFeign + Sentinel | 把博客拆 3 个微服务 |
| 数据 | Sharding-JDBC 分库分表 | 博客 posts 表水平拆 4 库 |
| 大数据 | Kafka + Flink 流处理 | 实时统计文章 PV 出 Grafana |
| k8s | Helm Chart + ArgoCD | 把 compose 改成 k8s 部署 |
| 算法 | LeetCode 200 + 5 道系统设计 | 中等难度 80% 通过 |

### 半年路线 (建立壁垒)

- 主导一个 5-10 人小组项目, 不再做 IC
- 写 5 篇深度技术文 (掘金 / 个人博客), 形成搜索引擎可见的技术品牌
- 看 2 本经典: 《DDIA》《SRE: How Google Runs Production Systems》
- 贡献 1 个开源项目 PR (Spring Boot / RocketMQ / 任何用过的)

### 1 年路线 (拿 P6/P7 offer)

- 简历: 1 个能讲 30 分钟的核心项目 + 1 个 side project + 1 个开源
- 面试: 系统设计 8-10 题熟练 + 算法中等 90%
- 软实力: 至少 1 次跨组协作 / 1 次主导技术决策的故事
- 目标公司清单 + 内推渠道铺设

### 反面教训 (常踩)

```
❌ "啥都学一点" - 简历技术栈 30 个, 一个都不深 → 面试一问就漏
❌ "只学不练" - 看完 60 章理论, 项目从没认真做完 → 简历空
❌ "不写不讲" - 学了不写文章 / 不讲给别人 → 学得快忘得快
❌ "只刷算法" - 算法刷 1000 道, 没项目和系统设计 → 进不了好公司
❌ "等机会" - 找工作前 1 个月才准备 → 焦虑且效果差
```

**正面建议**:

```
✓ 项目 > 算法 > 系统设计 > 八股, 按这个顺序投入时间
✓ 每月写 1 篇技术文 (不求完美, 求持续)
✓ 每季度做 1 次模拟面试 (跟人 / 录音)
✓ 每年换 1 次目标公司清单 (调整方向)
✓ 永远保持一个 "正在做的 side project"
```
